# WEB-4C: Web月次集計とB案申請書の金額整合性調査・修正記録

作成日: 2026-05-08  
ステータス: **調査・修正完了 / clasp push 済 / commit & push 済 / 本番 deploy 未実施**

---

## 問題

WEB-4B で集計0円問題を修正した後、Web月次詳細とB案申請書Excelに5円の差分が発覚した。

| 金額 | B案 申請書 Excel | Web 月次詳細（来院ヘッダ） |
|---|---|---|
| 来院合計 | **4,363円** | 4,368円（+5円） |
| 窓口負担 | 1,310円 | 1,310円（一致） |
| 保険請求 | **3,053円** | 3,058円（+5円） |

---

## 調査結果

### データソース比較

| 経路 | データソース | 計算方法 |
|---|---|---|
| **B案 Excel**（正ルート） | `V3TR_buildTransferDataForMonth_` | 施術明細から月単位で再計算 |
| **Web 月次詳細**（修正前） | 来院ヘッダ `visitTotal` / `windowPay` / `claimPay` 合算 | 来院都度の候補金額を単純合計 |

### 来院明細の内訳（hirayamaka / 2026-04）

| 施術日 | 区分 | 来院ヘッダ合計 | needCheck | 備考 |
|---|---|---|---|---|
| 2026-04-01 | 初検 | ¥2,495 | — | — |
| 2026-04-19 | 再検 | ¥1,873 | 要確認 | 同月別ケース初回 初検抑制 |
| **合計** | | **¥4,368** | | |

B案転記データ: **¥4,363**（差分 **5円**）

### 根本原因

2つの計算経路で丸め位置・集計粒度が異なる。

| 経路 | 丸め | 粒度 |
|---|---|---|
| `saveVisit_V3` / `calcHeaderAmountsByVisitKey_V3_` | 来院1件ずつ per-visit | 来院時点の暫定算定 |
| `V3TR_buildTransferDataForMonth_` | 月合計で `V3TR_roundToUnit_` | 施術明細からの完全再計算 |

この差は構造的なものであり、どちらかが「間違い」ではない。  
ただし **保険申請書に載るのは B案（転記データ）の金額** であるため、
Web 確認画面もそれに合わせる必要がある。

---

## 修正内容

**対象ファイル:** `web-monthly-claim-detail.html`

**変更:** Step 1「申請書転記データを生成」成功後のハンドラに以下を追加：

1. KPI 集計欄（来院合計・窓口負担・保険請求）を転記データ金額で上書き
2. tfoot（合計行）も同様に更新
3. 「申請書転記データより更新済み（申請額）」という注記を集計カードに追加

```
修正前フロー:
  ページ表示 → KPI = 来院ヘッダ合算（4,368円）→ Step1実行 → result欄に4,363円
  （KPIと結果欄で数値が一致しない）

修正後フロー:
  ページ表示 → KPI = 来院ヘッダ合算（初期表示 4,368円）
  → Step1実行 → KPI = 転記データ（4,363円）+ 注記「申請額（転記データより更新済み）」
  （Step1後はKPIが申請額と一致する）
```

## 残構造的課題（本フェーズでは対応しない）

来院ヘッダの per-visit 候補金額と転記データの月合計が構造上ズレる問題は残存。  
対応策候補（将来フェーズ）:
1. `saveVisit_V3` / `saveVisitFromWeb_V3` の金額算定を `V3TR_buildTransferDataForMonth_` と統一する
2. または来院ヘッダ金額の「候補値」を明示的に `needCheck=TRUE` の間のみとして扱い、Step1実行後に更新する

現時点では「Step1を実行してから申請書生成」という運用フローで一致する。
